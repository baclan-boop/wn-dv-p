=========================================
ПОСЛЕДНИЕ ОБНОВЛЕНИЯ (раздел «Реклама» / «Контракты»)
=========================================

- НОВОЕ: «➕ Добавить контракт» теперь НЕ пишет контракт в таблицу сразу —
  форма создаёт ЗАЯВКУ, которая попадает в новую вкладку «🕒 На одобрении»
  (видна только Curator AD, Dep. Director, Лидеру, Администратору — «Curator
  AD и выше»). Там можно поправить ЛЮБОЕ поле заявки (цвет, времена, даты,
  текст, скидку, сотрудника) и нажать «✅ Одобрить» — только после этого
  контракт реально появляется в таблице «Контракты» (или «❌ Отклонить» —
  тогда заявка просто помечается отклонённой, в таблицу ничего не
  попадает). И при подаче заявки, и при одобрении сервер заново проверяет,
  свободны ли нужные слоты. Слот с автоматической служебной пометкой
  «Перенос с ЧЧ:ММ» (её ставит сама система, когда время объявления
  переносят на другой слот — см. столбец «Время переноса») считается
  СВОБОДНЫМ для новой заявки, даже если формально в нём есть текст.
- «➕ Добавить контракт» (то есть подача заявки) по-прежнему доступна ВСЕМ
  сотрудникам отдела рекламы (Advertising Department и выше), а не только
  Curator AD и выше.
- Колонки «Принял»/«Откинул» в Контрактах, поле «Кто принял контракт» в
  форме «Добавить контракт» и выбор сотрудника при назначении премии —
  теперь поиск по первым буквам имени (можно и вставить скопированные
  откуда-то имя и фамилию), а не обычный выпадающий список.
- Время показа в форме «Добавить контракт» можно печатать вручную в
  формате ЧЧ:ММ (двоеточие подставляется само), список остался как
  подсказка. Показываются только времена текущих суток — без «хвоста»
  расписания, уходящего за полночь.
- Убрано ограничение «контракт можно заполнять только начиная со
  следующего дня» — теперь можно указать любую дату начала (по
  умолчанию сегодня).
- В результатах формы «Добавить контракт» — кнопки «⧉» для копирования
  сумм БЕЗ пробелов-разделителей, чтобы сразу вставлять в казну и т.п.
- Таблица «Контракты»: разделитель на каждый час (как в исходной
  Google Таблице), подсветка строки при отправленном объявлении (галочка
  «Статус») видна ВСЕМ ролям, поле «К выплате» больше не показывает «0»,
  который приходилось стирать перед вводом.
- Рядом с формой «Добавить контракт» — «📋 Сводка для клиента»: текст
  объявления, период, время показа и сумма заказа одним блоком для
  копирования и отправки клиенту.
- Блок «⏰ Ближайшие объявления к отправке» закреплён вверху страницы
  при прокрутке вниз.
- Добавлена светлая тема (переключатель 🌙/☀️ в шапке сайта) — влияет
  на Панель управления, раздел «Реклама» и страницу входа; публичные
  страницы (Главная/Новости/Услуги/Состав/О нас) остаются в фирменном
  тёмном оформлении с фоновыми фото независимо от темы.
- Можно изменить собственный ник (имя/фамилию персонажа) — кнопка
  «✎» рядом с именем в шапке сайта (и в мобильном меню).

   git push -u origin main

   !!! Замени ТВОЙ_НИК на свой никнейм GitHub !!!
   GitHub попросит войти — нажми "Войти через браузер".

6. Обнови страницу github.com/ТВОЙ_НИК/weazel-news
   Ты должен увидеть файлы: server.js, package.json, папку public.

=========================================
(интерфейс обновлён — актуальная инструкция)

1. Зайди на neon.tech → нажми [Sign up]
   Выбери [Continue with Google]. Карта не нужна.

2. После входа нажми [New project] (или [Create a project])

3. ПОЯВИТСЯ ОКНО "Create project" — заполни так:
   - Project name:    weazel-news
   - Postgres version: 18 (оставь как есть)
   - Region:          AWS Europe Central 1 (Frankfurt)
     (выбери из списка — это ближайший к EU регион)
   - Enable Neon Auth: НЕ включай (переключатель должен быть СЕРЫМ)
   - Нажми кнопку [Create] в правом нижнем углу

   Нажми на вкладку [Connection string] или найди секцию
   "Connection Details" / "Connect"

5. ПОЛУЧИТЬ СТРОКУ ПОДКЛЮЧЕНИЯ:
   - Найди выпадающее меню где написано "psql" или тип подключения
   - Переключи на: "Connection string" или "Prisma" или просто скопируй строку
   - Нужна строка которая НАЧИНАЕТСЯ С: postgresql://
   
   Она выглядит примерно так:
   postgresql://weazel_owner:ПАРОЛЬ@ep-xxx-xxx.eu-central-1.aws.neon.tech/neondb?sslmode=require

   ЕСЛИ НЕ ВИДИШЬ эту строку — смотри ниже:

   АЛЬТЕРНАТИВНЫЙ СПОСОБ получить строку:
   a) На странице проекта нажми кнопку [Connect] (обычно вверху)
   b) Появится окно — там будет вкладка [Connection string]
   c) Убедись что выбран тип: "Direct connection" или "Pooler"
   d) Скопируй строку начинающуюся с postgresql://

   СОХРАНИ ЭТУ СТРОКУ — понадобится на шаге 3!

6. Название базы данных:
   В новом интерфейсе Neon автоматически создаёт базу
   с именем "neondb". Это нормально, наш сервер использует
   любое имя — главное правильная строка подключения.

ШАГ 5: КАК ОБНОВИТЬ САЙТ (БЕЗ ПОТЕРИ ДАННЫХ)
=========================================

ГЛАВНОЕ ЧТО НАДО ПОНИМАТЬ:

  GitHub  → хранит только КОД (server.js, index.html и т.д.)
  Render  → запускает этот код
  Neon    → хранит ВСЕ ДАННЫЕ (новости, пользователи, настройки)

Это три РАЗНЫЕ вещи. Когда ты обновляешь код на GitHub —
Neon вообще не трогается. Новости, аккаунты и настройки
остаются на месте автоматически. Удалить их можно только
вручную через Neon SQL Editor или удалив сам проект на Neon.

Поэтому обновлять код можно сколько угодно раз — данные в
безопасности, если ты не трогаешь Neon напрямую.

--- КАК ПРАВИЛЬНО ОБНОВИТЬ КОД ---

1. Замени файлы в папке C:\\weazel-news\\
   (например server.js или public\\index.html)

   ВАЖНО: НЕ трогай файл .env если он у тебя есть локально —
   он не загружается на GitHub (это нормально, см. .gitignore)

2. Открой командную строку:
   cd C:\\weazel-news
   git add .
   git commit -m "обновление"
   git push

3. Render автоматически увидит изменения на GitHub и начнёт
   пересборку (1-3 минуты). Следи в разделе [Logs] на Render.

4. Когда сервер запустится, он САМ проверит структуру базы
   данных на Neon и при необходимости добавит новые таблицы/
   колонки (если они появились в новой версии кода) — но
   НИКОГДА не удаляет существующие данные. Это называется
   "безопасная миграция" (CREATE TABLE IF NOT EXISTS).

5. Открой сайт и убедись что новости/пользователи на месте.

--- ЧЕГО НЕЛЬЗЯ ДЕЛАТЬ (иначе данные пропадут) ---

✗ НЕ меняй переменную DATABASE_URL на Render на новую/другую —
  это переключит сайт на ДРУГУЮ базу данных (пустую).
  Меняй только если специально хочешь начать с чистого листа.

✗ НЕ удаляй проект на Neon.

✗ НЕ выполняй в Neon SQL Editor команды вроде:
  DROP TABLE news;  DELETE FROM news;  TRUNCATE news;
  (если только сам не хочешь удалить новости)

✗ НЕ создавай новый проект на Render "с нуля" под старый
  GitHub репозиторий, если хочешь сохранить данные —
  лучше обновлять СУЩЕСТВУЮЩИЙ сервис на Render, он сам
  подтянет новый код по git push.

--- РЕЗЕРВНАЯ КОПИЯ (на всякий случай, бесплатно) ---

Перед большими изменениями можно сделать мгновенный бесплатный
бэкап через функцию Neon Branches (входит в бесплатный план,
до 10 веток на проект):

1. Зайди в свой проект на neon.tech
2. В левом меню найди [Branches]
3. Нажми [Create branch]
4. Назови например: backup-2024-01-15
5. Нажми [Create]

Это создаёт мгновенную копию ВСЕЙ базы данных (новости,
пользователи, всё). Если что-то пойдёт не так после обновления
кода — можно открыть эту ветку и скопировать данные обратно,
либо переключить DATABASE_URL на эту ветку временно.

Удалить старые backup-ветки можно в любой момент — они не
занимают место сверх лимита, пока не растут отдельно от
основной базы.

ВОЗМОЖНЫЕ ПРОБЛЕМЫ
=========================================

ПРОБЛЕМА: Сайт не отвечает 30-60 секунд при первом заходе
РЕШЕНИЕ:  Нормально. Бесплатный Render засыпает после
          15 минут неактивности. Первый посетитель разбудит.

ПРОБЛЕМА: "ОШИБКА: DATABASE_URL не задан"
РЕШЕНИЕ:  Проверь Environment Variables на Render.
          Строка должна НАЧИНАТЬСЯ с: postgresql://

ПРОБЛЕМА: Не вижу строку подключения на Neon
РЕШЕНИЕ:  На странице проекта Neon:
          1. Нажми кнопку [Connect] вверху
          2. В окне выбери вкладку [Connection string]
          3. Или попробуй: Settings → Connection Details

ПРОБЛЕМА: "SSL connection required" или ошибка SSL
РЕШЕНИЕ:  В строке подключения должно быть: ?sslmode=require
          Если нет — добавь в конец строки вручную.

ПРОБЛЕМА: Забыл пароль администратора
РЕШЕНИЕ:  На Neon → SQL Editor → введи и выполни:
          DELETE FROM users WHERE email='computer52552@gmail.com';
          Потом на Render нажми [Manual Deploy] — admin создастся заново.

ПРОБЛЕМА: Случайно создал ВТОРОЙ сервис на Render для того же
          сайта, и непонятно какой из них правильный
РЕШЕНИЕ:  Открой оба сервиса на Render → Environment →
          посмотри какой DATABASE_URL у каждого. Тот что
          совпадает со строкой из Neon — рабочий, в нём твои
          данные. Второй (дубликат) можно удалить:
          Settings → Delete Web Service (это удалит только
          САМ СЕРВИС НА RENDER, база данных на Neon не пострадает).

ПРОБЛЕМА: После обновления кода новости/пользователи пропали
РЕШЕНИЕ:  99% случаев — это означает что DATABASE_URL изменился
          (например указывает на новый/другой проект Neon).
          Проверь Environment Variables на Render → DATABASE_URL
          должен быть ТОТ ЖЕ что и раньше. Если делал backup-ветку
          (см. ШАГ 5) — переключись на неё через Neon Branches.

ВАЖНО: файл google-apps-script/Code.gs — ОСОБЫЙ. Он не отправляется
       на GitHub/Render вместе с остальным кодом и не обновляется
       через git push. Он живёт отдельно, внутри самой Google Таблицы
       (Расширения → Apps Script). Если я пришлю тебе обновлённую
       версию этого файла — её нужно вручную скопировать в редактор
       Apps Script и опубликовать новую версию (см. ШАГ 6 выше).

=========================================
ШАГ 6: ПОДКЛЮЧИТЬ GOOGLE ТАБЛИЦУ (для панели «Объявления»)
=========================================

Панель «Объявления» ищет свободные слоты в твоей Google Таблице
(листы вида "04.07", "05.07" и т.д.) для роли Advertising Department.
Без настройки ниже панель покажет ошибку "Google Таблица не подключена".

Никакой сервисный аккаунт и доступ к Google Cloud НЕ нужен —
скрипт живёт прямо внутри самой таблицы, ты полностью его контролируешь.

--- Вставка скрипта в таблицу ---

1. Открой свою Google Таблицу с расписанием объявлений
2. В меню сверху: Расширения → Apps Script
   (откроется отдельная вкладка с редактором кода)
3. В редакторе будет файл Code.gs с заглушкой вида:
     function myFunction() { }
   Выдели всё (Ctrl+A) и удали.
4. Открой файл google-apps-script/Code.gs из этого архива,
   скопируй ВЕСЬ его текст и вставь в редактор Apps Script
5. Нажми значок 💾 сверху (или Ctrl+S) — сохранит проект
   (при первом сохранении попросит дать имя проекту — впиши любое)

--- Публикация как веб-приложения ---

1. В правом верхнем углу редактора нажми синюю кнопку [Deploy] /
   [Развернуть] → [New deployment] / [Новое развёртывание]
2. Рядом с "Select type" нажми на шестерёнку ⚙ → выбери [Web app] /
   [Веб-приложение]
3. Заполни:
   - Description: можно оставить пустым или написать "ads search"
   - Execute as: Me (твой аккаунт)
   - Who has access: Anyone (Все)
     ВАЖНО: без этого пункта сайт не сможет получить доступ к скрипту
4. Нажми [Deploy] / [Развернуть]
5. Google попросит разрешить доступ — нажми [Authorize access] →
   выбери свой аккаунт → на предупреждении "Google hasn't verified
   this app" нажми [Advanced] / [Дополнительные] → [Go to (название
   проекта), unsafe] — это нормально, это твой собственный скрипт
6. После этого появится ссылка вида:
     https://script.google.com/macros/s/AKfycb.../exec
   СКОПИРУЙ ЭТУ ССЫЛКУ ПОЛНОСТЬЮ — она понадобится дальше.

--- Добавление переменной на Render ---

На Render: твой сервис → Environment → добавь:

  KEY                      | VALUE
  --------------------------|---------------------------------------------
  GOOGLE_APPS_SCRIPT_URL   | ссылка, скопированная на предыдущем шаге

Нажми [Save Changes] — Render сам перезапустит сервис. Готово —
панель «Объявления» должна заработать сразу.

--- Если позже изменишь код скрипта ---

Просто сохранить (💾) недостаточно — изменения не попадут в уже
опубликованную ссылку, пока не создашь новую версию:
1. [Deploy] → [Manage deployments] / [Управление развёртываниями]
2. Нажми ✏️ (карандаш) рядом с активным развёртыванием
3. Version → выбери [New version] / [Новая версия]
4. [Deploy] / [Развернуть]
Ссылка остаётся той же самой — переменную на Render менять не нужно.

--- Важно про структуру таблицы ---

Скрипт по умолчанию ожидает такую структуру (как в примере, который
ты присылал):
- Листы (вкладки) названы датами в формате ДД.ММ, например "04.07"
- Зелёные объявления (Weazel News): колонка B — время, колонка D — текст
- Красные объявления (ADV): колонка K — время, колонка M — текст
- Данные начинаются с 3 строки (первые 2 строки — заголовки)
- Слот считается свободным, если ячейка в колонке "Текст" пустая

Если у тебя иначе — открой Code.gs, в самом верху есть блок
"НАСТРОЙКИ" с понятными названиями (GREEN_TIME_COL, RED_TEXT_COL
и т.д.) — поменяй числа под свою таблицу, сохрани, и опубликуй
новую версию (см. выше).

Скрипт также сам пробует несколько распространённых вариантов
названия листа для каждой даты (например и "04.07", и "4.7",
и "04/07", и "04.07.2026") — так что небольшие отличия в формате
(ведущие нули, слэш вместо точки, наличие года) не помешают.

--- Проверка что всё настроено верно (диагностика) ---

Открой ссылку веб-приложения (.../exec) ПРЯМО В БРАУЗЕРЕ, без
каких-либо параметров после неё. Вместо поиска слотов скрипт
покажет диагностику в формате JSON:
  - allSheetNames — список ВСЕХ реальных листов твоей таблицы
  - expectedTabNamesNext3Days — какие названия скрипт ожидает
    для ближайших 3 дней
  - todayUsedByScript — какую дату скрипт считает "сегодня"

Сравни allSheetNames с expectedTabNamesNext3Days — если листы с
нужными датами есть в allSheetNames, но их названия совсем не
похожи на expectedTabNamesNext3Days (например у тебя листы
называются "День 1", "Понедельник" и т.п., а не датами) — напиши
мне, поправлю логику под твою именно систему названий.

--- Если названия листов "визуально задваиваются" ---

В диагностике есть поле visualDuplicates. Если оно непустое —
значит в таблице есть листы, которые ВЫГЛЯДЯТ одинаково (например
два "05.07"), но технически это разные строки — Google Sheets не
разрешает создать два листа с побайтово одинаковым именем, поэтому
если два листа выглядят одинаково, между ними обязательно есть
невидимая разница: лишний пробел на конце, неразрывный пробел
вместо обычного, или похожий на точку символ вместо настоящей точки.

Это частая причина, почему поиск "не находил слоты", даже если
всё остальное настроено верно — скрипт искал точное совпадение
и не находил его.

Начиная с этой версии скрипт сам это учитывает: он сравнивает
названия ПОСЛЕ очистки от пробелов и похожих символов, и если
находит несколько подходящих листов — берёт более новый (правый
по порядку вкладок). Обновлять код не нужно только если ты уже
скопировал последнюю версию Code.gs из этого архива.

--- Если видишь ошибку "Google вернул страницу входа" ---

Это значит развёртывание опубликовано с неверными правами доступа.
Исправляется так:
1. [Deploy] / [Развернуть] → [Manage deployments] / [Управление
   развёртываниями]
2. Нажми ✏️ (карандаш) рядом с развёртыванием
3. Проверь: "Who has access" / "У кого есть доступ" должно быть
   ИМЕННО "Anyone" / "Все" (не "Only myself", не "Anyone with
   Google account")
4. Если было неверно — исправь, затем Version → [New version] →
   [Deploy] / [Развернуть]

--- Необязательно: часовой пояс ---

Google Apps Script по умолчанию считает "сегодня" в UTC. Если день
должен начинаться по другому времени — открой Code.gs, в блоке
"НАСТРОЙКИ" найди TZ_OFFSET_HOURS и поставь нужный сдвиг
(например 3 для UTC+3), затем сохрани и опубликуй новую версию.

--- О безопасности этого способа ---

Ссылка на веб-приложение с доступом "Anyone" технически может быть
вызвана кем угодно, кто её узнает — но узнать её со стороны нельзя
(она хранится только в переменной окружения на Render, не в коде
сайта и не в браузере). Через неё можно получить только список
свободных дат/времени — никаких имён клиентов, цен или текстов
объявлений скрипт наружу не отдаёт. Доступ к самой панели на сайте
по-прежнему требует роли Advertising Department или Администратор.

=========================================
ШАГ 7: НОВОЕ — РОЛИ, КОНТРАКТЫ, КАЛЬКУЛЯТОР, СТАТИСТИКА
=========================================

Ничего вручную настраивать не нужно — при первом запуске обновлённого
кода сервер сам создаст новые таблицы в базе данных (Neon это не
затронет твои существующие новости/пользователей).

--- Роли и права доступа ---

  1. Администратор — полный доступ ко всему без исключений.

  2. Лидер — доступ как у Администратора, КРОМЕ вкладок «Посетители»
             (статистика посещений) и «Логи» (журнал редактирования) —
             это зона видимости исключительно Администратора. В
             Пользователях может назначать только «средние» роли (Dep.
             Director/Редактор/Curator AD/Advertising Department/Гость) —
             не может назначить или отобрать роль Leader/Admin (см. ниже).

  3. Dep. Director — доступ как у Лидера (в т.ч. НЕ видит «Посетители» и
             «Логи»), но дополнительно НЕ может:
               • управлять Составом (вкладка «Состав» скрыта);
               • менять «Все тексты» сайта и «Фоны страниц» (обе вкладки
                 скрыты).
             В разделе «Реклама» — ПОЛНЫЙ доступ наравне с Curator AD:
             все поля Контрактов, Статистика, Премии, добавление
             контрактов. В Пользователях — те же ограничения по
             назначению ролей, что и у Лидера (см. п. 2 выше), НО с
             ОДНИМ дополнительным исключением: саму роль Dep. Director
             назначить не может (даже другому Dep. Director) — эту роль
             выдают только Лидер и Администратор.

  4. Редактор — доступ только к вкладке «Новости»: может добавлять и
             редактировать новости, но НЕ может их удалять. Не имеет
             доступа ни к разделу «Реклама», ни к каким другим вкладкам
             Панели (Услуги, Состав, Пользователи, Сотрудники, Тексты,
             Настройки, Посетители, Логи).

  5. Curator AD — в разделе «Реклама» ПОЛНОЕ право редактирования: все
             поля Контрактов, Статистика, Премии, добавление контрактов.
             Дополнительно в Панели управления видит две вкладки:
               • «Сотрудники» — полное управление ростером (добавление/
                 редактирование/удаление/активация);
               • «Пользователи» — но ограниченно: может выдавать и
                 снимать ИСКЛЮЧИТЕЛЬНО роль Advertising Department (и
                 только у пользователей, у которых сейчас Гость либо уже
                 Advertising Department — более высокие роли ей
                 недоступны).
             Остальные вкладки Панели (Новости, Услуги, Состав, Тексты,
             Настройки, Посетители, Логи) не видит.

  6. Advertising Department — доступ к Калькулятору и к поиску свободных
             слотов в Google Таблице (см. ШАГ 6), а также к Контрактам
             и Статистике — но ТОЛЬКО НА ПРОСМОТР большей части полей.
             В Контрактах им можно менять статус (галочку), «Откинул»
             и время переноса, но НЕЛЬЗЯ менять текст, цену контракта,
             «Принял» и сумму к выплате (эти поля показаны
             заблокированными). Вкладка «➕ Добавить контракт» ИМ ТОЖЕ
             доступна — заполнение контракта одной формой открыто всем
             сотрудникам отдела рекламы. В Статистике им видны таблица
             и уже назначенные премии, но без кнопок «+ Премия» /
             «Выписать премии» и без возможности отметить премию
             выплаченной или удалить её — это управляется только
             Curator AD и выше.

  7. Гость — обычный авторизованный посетитель, без прав.

Кто какую роль может НАЗНАЧАТЬ другому пользователю (вкладка Панель →
Пользователи):
  • Администратор — любую роль любому пользователю, без ограничений.
  • Лидер — «средние» роли: Dep. Director, Редактор, Curator AD,
    Advertising Department, Гость. Роль Leader или Admin назначить не
    может, и не может менять роль пользователю, который сейчас Leader
    или Admin.
  • Dep. Director — то же самое, что и Лидер, но БЕЗ роли Dep. Director:
    Редактор, Curator AD, Advertising Department, Гость. Роль
    Dep. Director (как и Leader/Admin) назначить не может никому.
  • Curator AD — исключительно роль Advertising Department (может
    выдать её Гостю и снять обратно до Гостя); других ролей не
    касается.
  • Редактор и Advertising Department — ролями не управляют вообще,
    вкладка «Пользователи» им не видна.
Кнопка «Роль» в списке пользователей видна только тем, у кого есть
право изменить роль конкретно этого пользователя; в самом выпадающем
списке недоступные варианты показаны заблокированными. Вкладка
«Пользователи» видна Dep. Director, Лидеру, Администратору и Curator AD.

--- Новые разделы сайта (одна кнопка «Реклама» в шапке, дальше — вкладки) ---

В шапке сайта теперь ОДНА кнопка «Реклама» (видна всем ролям, у кого
есть доступ хотя бы к одному из разделов ниже). Внутри — вкладки:
«Объявления», «Калькулятор», «Контракты», «Статистика». Показываются
только те вкладки, на которые есть права у текущей роли.

• «Объявления» — поиск свободных слотов в Google Таблице (см. ШАГ 6).

• «Калькулятор» — расчёт стоимости контракта: вводишь текст объявления,
  кол-во объявлений в день, кол-во дней, скидку — считает сумму заказа,
  долю казны (90%), долю сотрудника (10%) и выплату за 1 объявление.
  Отдельно для 🟢 зелёных (× 300 за символ) и 🔴 красных/СМС (× 150).

• «Контракты» — интерактивная таблица объявлений на выбранный день
  (аналог твоей Google Таблицы, но прямо на сайте): статус, время,
  цена, текст, принял/откинул (из ростера сотрудников), к выплате,
  время переноса. При вводе времени переноса двоеточие подставляется
  автоматически (ЧЧ:ММ). Текст исходной строки (реальный текст
  контракта) при этом НЕ меняется — вместо этого колонка «Текст» в
  СТРОКЕ ТОГО ВРЕМЕНИ, НА КОТОРОЕ перенесли, автоматически заполняется
  пометкой «Перенос с ЧЧ:ММ» (где ЧЧ:ММ — исходное время слота).
  Например: в строке 13:00 указали время переноса 13:20 — тогда в
  строке 13:20 в колонке «Текст» появится «Перенос с 13:00», а текст
  самой строки 13:00 останется прежним. Если в целевой строке уже
  был какой-то другой текст — перед перезаписью появится запрос на
  подтверждение. Каждое поле сохраняется сразу же по мере ввода.
  Доступно Curator AD, Dep. Director, Лидеру, Администратору (полный
  доступ), а также Advertising Department (просмотр + статус/«Откинул»/
  время переноса).

  Внутри «Контрактов» — третья вкладка «➕ Добавить контракт»: вместо
  ручного заполнения ячейка за ячейкой, можно одной формой оформить
  весь контракт разом. Указываешь тип (🟢/🔴), время(-ена) показа в
  день (можно добавить несколько — кнопка «+ Ещё время», время можно
  печатать вручную в формате ЧЧ:ММ, а не только выбирать из подсказок),
  срок в днях, текст объявления, скидку и кто принял контракт (поиск
  сотрудника по первым буквам имени — можно и вставить скопированные
  откуда-то имя и фамилию). Сервер САМ проверяет, свободны ли нужные
  времена на все нужные дни — если хоть один слот уже занят, ничего не
  создаётся, и показывается список именно занятых дат/времён. Если всё
  свободно — одним действием проставляется текст, сотрудник в колонку
  «Принял», и автоматически считаются цена и выплата за 1 объявление
  (по той же формуле, что и в Калькуляторе, с учётом скидки). Дата
  начала — любая на выбор (по умолчанию сегодня); прежнее жёсткое
  ограничение «только начиная со следующего дня» снято — из-за него
  не получалось занести контракт, принятый поздно вечером, если факт
  внесения в систему уже пришёлся на время после полуночи. Рядом с
  формой — «📋 Сводка для клиента»: текст объявления, период, время
  показа и сумма заказа одним блоком, который можно скопировать и
  отправить клиенту.

  В само поле «Текст объявления» формы вводится обычный текст (без
  каких-либо команд) — именно по его длине (в символах) считается
  цена. А вот в колонку «Текст» таблицы «Контракты» на нужные
  дату(-ы)/время(-на) этот текст попадает уже с префиксом команды
  «/wnews » впереди — чтобы его можно было сразу скопировать и
  вставить в игровой чат. Например, если в форме ввести «Идёт набор
  в стрелковую семью Hinode. Почта для связи: xuwhuk@ds.gg,
  bobidig@ds.gg.», то в таблице появится «/wnews Идёт набор в
  стрелковую семью Hinode. Почта для связи: xuwhuk@ds.gg,
  bobidig@ds.gg.». Префикс не учитывается при расчёте цены/выплаты —
  он добавляется только к тому, что сохраняется в таблицу.

• «Статистика» — вкладки «Текущая неделя» / «Прошлая неделя»
  (понедельник—воскресенье), считаются автоматически из таблицы
  контрактов. Строка сотрудника подсвечивается красным, если «К
  выплате» меньше 15,000$ (эта сумма считается по слотам, где
  сотрудник стоит в «Принял»). Ниже — блок «Премирование» для
  назначения бонусов вручную (кнопка «+ Премия», только у Curator AD
  и выше). Более старые недели не удаляются — их можно посмотреть
  через поле «🔎 Архив», указав любую дату внутри нужной недели.

  Кнопка «📋 Выписать премии» (тоже только у Curator AD и выше)
  автоматически формирует список премий за просматриваемый период:
  для каждого сотрудника берётся ТА ЖЕ сумма, что показана у него в
  колонке «К выплате» в таблице статистики выше (то есть по слотам,
  где он стоит в «Принял», за этот период) — это и есть его премия.
  В список попадают ВСЕ сотрудники, задействованные за период
  (принявшие или откинувшие хоть один слот), по одному на строку,
  в формате:

    StaticID;сумма;Выплаты AD

  без пробелов, через точку с запятой — например:
  «45058;4560;Выплаты AD». Готовый список можно скопировать одной
  кнопкой и вставить куда нужно. Сотрудники без указанного StaticID
  в список не попадают — об этом покажется предупреждение.

• Панель → «Сотрудники» (доступно Curator AD, Dep. Director, Лидеру,
  Администратору) —
  ростер отдела рекламы (имя персонажа + StaticID), используется в
  выпадающих списках «Принял/Откинул» в Контрактах, в форме «Добавить
  контракт» и в Статистике. Добавь сюда всех сотрудников, которые должны
  появляться в этих списках (только «активные» сотрудники показываются
  в форме «Добавить контракт»).

--- Если нужно изменить время работы таблицы контрактов ---

По умолчанию слоты идут с 13:00 до 03:00 (следующего дня) с шагом
10 минут — как в присланном примере. Если нужно другое расписание,
это можно поменять без правки кода, отправив PUT-запрос на /api/settings
с телом {"contractSchedule":{"start":"13:00","end":"03:00","intervalMin":10}}
(от имени Лидера/Администратора — только у них есть право на PUT
/api/settings) — например через консоль браузера:
  fetch('/api/settings',{method:'PUT',headers:{'Content-Type':'application/json'},
  credentials:'include',body:JSON.stringify({contractSchedule:{start:'13:00',end:'03:00',intervalMin:10}})})

=========================================
ШАГ 8: ПОДКЛЮЧИТЬ CLOUDINARY (чтобы фоны, аватарки и картинки
новостей не пропадали при обновлении сайта)
=========================================

ПОЧЕМУ ЭТО ВАЖНО:

  Когда ты загружаешь картинку через сайт (фон страницы, аватар
  участника в «Составе», картинку к новости) — файл раньше сохранялся
  прямо на диск Render. Но Render (как и большинство бесплатных
  хостингов) при каждом обновлении кода (redeploy) пересоздаёт диск
  контейнера с нуля — а значит ВСЕ загруженные так картинки исчезают,
  хотя в базе Neon ссылка на них остаётся (получается "битая" картинка).

  Cloudinary — бесплатный облачный хостинг именно для картинок.
  Файл, загруженный туда, живёт независимо от Render и не пропадает
  ни при каком redeploy. Сервер теперь загружает КАЖДУЮ картинку,
  которую загружают через сайт, сразу в Cloudinary — для этого нужно
  всего 3 переменные окружения.

--- КАК ПОЛУЧИТЬ ДАННЫЕ CLOUDINARY ---

1. Зайди на cloudinary.com → [Sign up for free] (можно через Google)
2. После входа откроется Dashboard — на нём сразу видны 3 нужных
   значения (раздел "Product Environment Credentials" / "API Keys"):
   Cloud name, API Key, API Secret.

--- КАК ДОБАВИТЬ ИХ НА RENDER ---

1. Открой свой сервис на render.com → вкладка [Environment]
2. Нажми [Add Environment Variable] и добавь ПО ОДНОЙ:

   KEY                     | VALUE
   ========================|========================================
   CLOUDINARY_CLOUD_NAME   | (Cloud name из Cloudinary Dashboard)
   CLOUDINARY_API_KEY      | (API Key из Cloudinary Dashboard)
   CLOUDINARY_API_SECRET   | (API Secret из Cloudinary Dashboard)

3. Нажми [Save Changes] — Render сам перезапустит сервис.
4. В логах при старте должна появиться строка:
   "Cloudinary подключён — загруженные файлы переживут редеплой"

БЕЗ этих трёх переменных сайт продолжит работать как раньше (просто
будет по-старому сохранять файлы локально на Render — то есть фоны,
аватарки и картинки новостей снова будут пропадать при обновлении).
